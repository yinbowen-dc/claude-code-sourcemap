# 集群 / 多 Agent 系统

## 概述

Swarm（集群）系统允许 Claude Code 并行运行多个 Claude 实例（"队友"），
由一个"领导者"实例协调。这是整个代码库中架构最复杂的子系统之一。

**相关文件：** `restored-src/src/utils/swarm/`（18 个文件）+ `utils/teammate*.ts`

## 后端抽象层

```
TeammateExecutor（接口）
  ├── InProcessBackend    — 同一 Node.js 进程内，AsyncLocalStorage 隔离
  ├── PaneBackendExecutor — 可视化面板后端的适配器
  │     ├── TmuxBackend   — 通过 tmux split-window 管理外部面板
  │     └── ITermBackend  — 通过 iTerm2 Python API（it2 CLI）管理原生面板
  └── （运行时通过 registry.ts 自动检测）
```

### 后端选择优先级（registry.ts）

```
1. 在 tmux 内？          → TmuxBackend（始终优先）
2. 在 iTerm2 内？
   a. 未设置 preferTmux 且 it2 CLI 可用 → ITermBackend
   b. tmux 可用 → TmuxBackend（native: false，needsIt2Setup: true）
   c. 两者都不可用 → 报错
3. 非交互式会话（--print 模式） → InProcessBackend
4. tmux 可用 → TmuxBackend（外部会话）
5. 报错：显示安装说明
```

检测结果**在会话生命周期内缓存**（运行中环境不会改变）。

### 队友模式快照

`teammateModeSnapshot.ts` 在启动时捕获配置模式并冻结。
防止运行时配置变更导致会话中途切换模式。

```typescript
export type TeammateMode = 'auto' | 'tmux' | 'in-process'

// 在 main.tsx 早期调用一次
captureTeammateModeSnapshot()

// 后续读取始终返回冻结的值
getTeammateModeFromSnapshot(): TeammateMode
```

在 `'auto'` 模式下：如果某次生成回退到进程内模式，`markInProcessFallback()` 会
让这个选择在整个会话中"粘住"，不再重新检测。

## 后端详情

### TmuxBackend（tmux 后端）

- **在 tmux 内部：** 分割领导者窗口（领导者占左 30%，队友占右 70%，N>1 时使用二叉树布局）
- **在 tmux 外部：** 用外部 socket 创建独立的 `claude-swarm-{PID}` 会话
- **隐藏/显示：** `break-pane` 将面板移到隐藏会话；`join-pane` 恢复
- **Socket 命名：** `claude-swarm-${process.pid}` — 按 PID 命名防止冲突

**N 个队友的二叉树布局：**
```
teammate_count % 2 === 1  → 垂直分割
否则                      → 水平分割
目标面板 = floor((count-1)/2)
```

分割后等待 200ms（`PANE_SHELL_INIT_DELAY_MS`），等 shell rc 文件加载完毕再发送命令。

### ITermBackend（iTerm2 后端）

使用 `it2` CLI（iTerm2 Python API 包装器）。

- **会话追踪：** 存储 `teammateSessionIds[]` 以便后续分割时定位面板
- **并发生成安全：** 互斥锁风格的 `paneCreationLock` Promise 链：

```typescript
let paneCreationLock: Promise<void> = Promise.resolve()

function acquirePaneCreationLock(): Promise<() => void> {
  let release: () => void
  const newLock = new Promise<void>(resolve => { release = resolve })
  const previousLock = paneCreationLock
  paneCreationLock = newLock
  return previousLock.then(() => release!)
}
```

- **自愈重试：** 如果分割时目标会话已死，通过 `session list` 确认后删除 ID 并重试。
  最多迭代 N 次（每次删除一个死亡 ID）。

**检测陷阱：** 使用 `it2 session list`（而非 `--version`），因为即使 Python API 被禁用，
`--version` 仍会返回成功。

### InProcessBackend（进程内后端）

- 在同一进程内以 **AsyncLocalStorage 隔离上下文**运行队友
- 传给队友前**清空**父进程的消息历史：
  ```typescript
  toolUseContext: { ...this.context, messages: [] }
  ```
  防止父会话对话被固定在内存中占用队友生命周期。
- 优雅终止：向邮箱发送 shutdown-request 消息
- 强制终止：调用 `AbortController.abort()`

## 队友通信机制

### 基于文件的邮箱

```
~/.claude/teams/{teamName}/mailbox/{agentName}/  ← 每个 Agent 的收件箱
```

消息类型：
- `permission_request` / `permission_response`（权限请求/响应）
- `shutdown_request`（关闭请求）

### 权限同步（permissionSync.ts）

**两代权限 IPC 机制：**

**旧版（基于文件）：**
```
~/.claude/teams/{teamName}/permissions/pending/{requestId}.json
~/.claude/teams/{teamName}/permissions/resolved/{requestId}.json
```
文件级 `.lock` 防止并发读写。领导者轮询 pending 目录，
通过原子性地将文件移到 resolved/ 来完成响应。

**新版（基于邮箱）：**
领导者直接向工作者邮箱写入 JSON 消息作为响应。
工作者从邮箱队列中读取。

### 领导者权限桥（leaderPermissionBridge.ts）

进程内队友无法显示自己的权限对话框，而是注入到**领导者的 React 状态**中：

```typescript
// 领导者注册对话框 setter（只注册一次）
registerLeaderToolUseConfirmQueue(setter: SetToolUseConfirmQueueFn)

// 队友调用此函数在领导者 UI 中显示权限提示
getLeaderToolUseConfirmQueue()?.((prev) => [...prev, newPrompt])
```

### 团队文件

```
~/.claude/teams/{teamName}/team.json
```

包含 `leadAgentId`、`members[]`（名称、agentId、颜色）、创建时间戳。
用于重连：会话恢复时从此文件重建团队成员信息。

## 反常规设计

| 模式 | 原因 |
|------|------|
| 副作用后端注册（文件末尾调用 `registerTmuxBackend(TmuxBackend)`） | 避免循环导入；注册表对后端实现透明 |
| `inProcessFallbackActive` 粘性标志 | 一旦选择进程内作为回退，后续不再重新检测 |
| 队友消息清空 | 不继承父会话对话，防止内存泄漏 |
| 二叉树面板布局 | 无需复杂布局算法即可实现符合人机工学的分块 |
| 面板 shell 初始化 200ms 延迟 | shell rc 文件需要加载时间；过早发送命令会丢失 |
| `getOrCreateWorktree` 快速路径：直接读取 `.git` 指针文件 | 避免 git 子进程调用（节省约 14ms） |
