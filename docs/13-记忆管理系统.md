# 记忆管理系统

## 概述

Claude Code 拥有完整的记忆管理体系，分为四层：用户长期记忆、会话反馈记忆、项目记忆、参考文档记忆。
通过 Sonnet 模型驱动的语义检索，在不同会话之间保持知识积累。

**相关文件：** `memdir/`、`services/extractMemories/`、`services/teamMemorySync/`、`services/SessionMemory/`

---

## memdir：内存文件系统

### 四种记忆类型

```
user/       用户通用记忆（跨项目持久化）
            存储路径：~/.claude/memory/user/
feedback/   用户对 Claude 行为的纠正与偏好
            存储路径：~/.claude/memory/feedback/
project/    项目专属记忆
            存储路径：~/.claude/projects/{hash}/memory/project/
reference/  用户手动提供的参考文档
            存储路径：~/.claude/projects/{hash}/memory/reference/
```

### Sonnet 语义检索器

```typescript
// 最多返回 5 条相关记忆
// 使用 claude-sonnet 模型进行语义匹配
const MAX_MEMORIES_TO_RETURN = 5

// 200 文件扫描上限（防止大型记忆库的性能下降）
const MAX_FILES_TO_SCAN = 200

// 检索流程：
// 1. 扫描所有 .md 记忆文件（最多 200 个）
// 2. 将文件内容 + 当前对话上下文发给 Sonnet
// 3. Sonnet 返回最相关的 ≤5 条记忆的文件名
// 4. 读取并注入到系统提示
```

### 尾部运行队列（Trailing Run Queue）

```typescript
// 解决记忆提取的并发问题
// 问题：记忆提取是异步的，对话可能在提取完成前继续
// 方案：尾部队列——只保留最新一次待处理提取请求
// 如果有新请求到达，丢弃旧的，用新的替换
class TrailingRunQueue {
  private pending: (() => Promise<void>) | null = null

  enqueue(fn: () => Promise<void>) {
    this.pending = fn  // 总是替换，不是累积
  }
}
```

## teamMemorySync：团队记忆同步

### 架构

```typescript
// 将团队共享记忆从远程仓库同步到本地
// 使用文件监视器（chokidar）监控本地变更
// 变更触发增量 delta 同步（非全量）
```

### 30+ 条密钥检测规则

团队记忆同步包含**密钥扫描器**，防止把密钥提交到共享记忆中：

```typescript
// 检测模式覆盖：
// - AWS 访问密钥（AKIA... 前缀）
// - GitHub 个人访问令牌（ghp_... / github_pat_...）
// - Anthropic API 密钥 ← 特别处理
// - OpenAI API 密钥（sk-...）
// - Stripe 密钥
// - Slack Bot Token
// - 私钥 PEM 块
// ... 共 30+ 种模式
```

### 🔥 Anthropic 密钥运行时拼接（绕过构建扫描）

```typescript
// Anthropic API 密钥前缀 "sk-ant-" 在源码中不直接出现
// 而是在运行时通过数组拼接构建：
const ANTHROPIC_KEY_PREFIX = ['sk', 'ant'].join('-') + '-'
// 原因：构建流水线扫描源码文件中的密钥模式
// 如果直接写 'sk-ant-'，会触发自己代码库的密钥扫描器
// 运行时拼接让检测器对自身代码库失效
```

### 永久失败抑制 Bug（已修复）

```typescript
// 历史 Bug：同步失败时进入永久失败状态，停止尝试
// 但文件监视器仍在运行，每次文件变更都触发失败回调
// 结果：167,000 次/天 无效 API 调用
// 修复：失败后完全停止文件监视器，而不仅是设置失败标志
```

## SessionMemory：会话记忆压缩

### 原理

```typescript
// 在对话压缩（/compact）时，不完全丢弃历史
// 而是将精华提炼成结构化摘要存储
// 下次会话或压缩后，摘要重新注入上下文

// 存储位置：项目配置文件中的 sessionMemory 字段
// 格式：带时间戳的 Markdown 摘要
```

### 工具对（Tool-Pair）分割修复

```typescript
// 历史 Bug：压缩边界落在工具调用和工具结果之间时
// 摘要只看到了工具调用，但没有工具结果
// 修复：确保工具对（tool_use + tool_result）总是一起保留
// 不允许压缩边界切断工具对
```

## 反常规设计

| 模式 | 原因 |
|------|------|
| Sonnet 做记忆检索（非向量数据库） | 小型代码库无需维护向量索引；Sonnet 的语义理解更准确 |
| 200 文件扫描上限 | 防止大型记忆库导致系统提示过长 |
| Fork 子进程提取 | 记忆提取不阻塞主交互循环 |
| 尾部运行队列 | 快速对话中只提取最新状态，避免积压 |
| Anthropic 密钥前缀运行时拼接 | 绕过自身构建流水线的密钥扫描器 |
| 永久失败修复 | 阻止文件监视器触发无效 API 调用循环（167K/天） |
| 工具对完整性保护 | 防止压缩边界切断工具调用和结果，导致上下文破损 |
