### 为何是 40+ useState 而非单一状态对象

```typescript
// 刻意的性能设计：
// 每个 useState 独立更新，不触发无关渲染
// 例如：isRunning 变化只重渲染依赖 isRunning 的组件
// 如果是单一对象 {isRunning, messages, ...}，
// 任何字段变化都会触发完整重渲染
```

## StreamingToolExecutor：并发工具执行

### 安全并发 vs 串行分区

```typescript
// 工具按并发安全性分区：

// 可并发工具（concurrentSafeTools）：
// - Read、Glob、Grep（只读）
// - WebFetch（网络读取）
// - Task（生成子 Agent，独立）
// 这些工具可以同时运行，互不干扰

// 必须串行工具（serialTools）：
// - Write、Edit（写文件，可能竞争同一文件）
// - Bash（有副作用，顺序可能影响结果）
// - 任何修改状态的工具

// 执行策略：
// 1. 先批量运行所有可并发工具
// 2. 等所有并发工具完成
// 3. 串行运行所有需要串行的工具
```

### 🔥 只有 Bash 错误触发级联取消

```typescript
// 工具执行错误处理是非对称的：

// Bash 工具错误：
// → 触发级联取消：取消所有正在运行的其他工具
// → 原因：Bash 命令失败通常意味着系统状态不一致
//         继续其他工具可能产生更严重的问题

// 其他工具错误（Read、Write、Edit 等）：
// → 只报告该工具的错误
// → 不影响其他并发运行的工具
// → 原因：文件操作失败通常是局部问题，不影响整体状态
```

### 流式输出聚合

```typescript
// 并发工具的输出需要聚合展示
// 策略：按工具调用 ID 分组，保持顺序一致

// 展示规则：
// - 每个工具调用有独立的输出区域
// - 多个工具同时流式输出时，分栏展示
// - 串行工具完成后，展示整合结果
```

## 反常规设计

| 模式 | 原因 |
|------|------|
| 4001 = 瞬态压缩重连 | 自定义 WebSocket 关闭码用于传递语义信息 |
| 合成消息 token=0 | 权限同步不是 API 调用，不应计入费用 |
| 40+ useState（非单一状态） | 细粒度重渲染控制；单一状态对象导致过度渲染 |
| 只有 Bash 触发级联取消 | Bash 错误影响系统全局状态；其他工具错误是局部的 |
| 工具结果持久化到文件 | 大结果消耗内存；文件存储可以被操作系统虚拟内存管理 |
| 并发/串行分区执行 | 最大化并行度同时保证写操作的一致性 |
