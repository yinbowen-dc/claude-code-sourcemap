# 工具与权限系统

## 概述

Claude Code 实现了 40+ 个内置工具以及动态 MCP 工具，所有工具都受多层权限系统控制。

**相关文件：** `restored-src/src/tools/` + `Tool.ts` + `tools.ts` + `utils/toolPool.ts` +
`utils/toolResultStorage.ts` + `utils/toolSearch.ts` + `hooks/useCanUseTool.tsx`

## 权限模式

```typescript
type PermissionMode =
  | 'default'           // 对危险操作询问用户
  | 'acceptEdits'       // 无需询问即接受文件编辑
  | 'plan'              // 仅分析，不执行
  | 'bypassPermissions' // 自动接受所有操作（YOLO 模式）
  | 'auto'              // 基于分类器的自动审批
  | 'dontAsk'           // 旧版：拒绝所有权限提示
```

切换方式：`default → acceptEdits → plan → bypassPermissions → auto → default`（Shift+Tab）

### 权限决策流程（useCanUseTool.tsx）

```
1. 检查中断信号
2. 有 forceDecision？→ 直接决定（绕过所有检查，用于测试）
3. hasPermissionsToUseTool(工具, 输入, 上下文)
     ├─ 'allow' → 记录 + 决定
     ├─ 'deny'  → 记录 + 决定
     └─ 'ask'
          ├─ 协调器工作者？→ handleCoordinatorPermission()
          ├─ Swarm 工作者？→ handleSwarmWorkerPermission()
          ├─ 2 秒宽限期：推测性 bash 分类器
          │    高置信度匹配 → 自动审批
          └─ 显示交互式权限对话框 → handleInteractivePermission()
```

### 规则来源（优先级从高到低）

1. `cliArg` — CLI 参数
2. `command` — 交互式 `/permissions allow …`
3. `session` — 会话级规则
4. `globalConfig` — 全局 `~/.claude/config.json`
5. `userConfig` — 用户设置
6. `workspaceConfig` — 工作区 `.claude/settings.json`

规则格式：`ToolName(pattern)`，如 `Bash(git *)`、`mcp__github`

## 工具池组装（toolPool.ts）

```typescript
assembleToolPool(permissionContext, mcpTools):
  1. 获取内置工具（遵守模式过滤）
  2. 按拒绝规则过滤 MCP 工具
  3. 按名称去重（内置工具优先）
  4. 排序：内置工具在前（字母序），然后 MCP（字母序）
```

确定性排序对**提示缓存稳定性**至关重要。工具顺序任何改变都会导致
约 11,000 Token 的缓存位置失效。

### 延迟工具加载（toolSearch.ts）

当 50+ 个 MCP 工具会超过 Token 预算时，工具被延迟加载，
通过 `ToolSearchTool` 按需加载：

```typescript
type ToolSearchMode =
  | 'tst'        // 始终延迟 MCP 工具（默认）
  | 'tst-auto'   // 仅在超过阈值时延迟（默认为上下文窗口的 10%）
  | 'standard'   // 不延迟（全部内联）

// 非 Anthropic 的 ANTHROPIC_BASE_URL 禁用（第三方网关通常不支持 tool_reference beta）
```

## 工具结果存储（toolResultStorage.ts）

当工具输出超过 `maxResultSizeChars`（默认 50,000 字符 / 256KB）时：

1. 完整输出保存到磁盘：`{projectDir}/{sessionId}/tool-results/{toolUseID}.{json|txt}`
2. 预览（前 2000 字节）以 `<persisted-output>` 标签包裹内联返回
3. 幂等性：`'wx'` 标志防止 API 重放时重复写入

持久化阈值可通过 GrowthBook 标志 `tengu_satin_quoll` 按工具覆盖。

## 权限绕过机制

### 1. `forceDecision` 参数
```typescript
// 由测试注入；跳过所有权限检查
async useCanUseTool(..., forceDecision?: PermissionDecision)
```

### 2. `bypassPermissions` 模式
Shift+Tab 可切换；跳过每一项检查：
```typescript
if (mode === 'bypassPermissions') return { behavior: 'allow', updatedInput: input }
```
可通过 GrowthBook `tengu_disable_bypass_permissions_mode` 全局禁用。

### 3. 推测性 Bash 分类器
匹配预分析安全模式的 bash 命令在 2 秒内跳过对话框。

### 4. 权限 Hooks
自定义 hooks 可在对话框出现之前预先批准操作。

### 5. 自动模式下危险规则剥离
进入 auto 模式时，匹配某些模式的规则（如 `Bash(*)`、`Bash(eval)`、`PowerShell(*)`、`REPL(*)`）
会被剥离——它们会绕过分类器安全检查。

## 工具定义合约（Tool.ts）

```typescript
type Tool = {
  name: string
  call(...): Promise<ToolResult>
  description(): Promise<string>
  prompt(): Promise<string>
  inputSchema: ZodType
  isEnabled(): boolean
  isConcurrencySafe(input): boolean   // 默认：false（安全失败）
  isReadOnly(input): boolean           // 默认：false（安全失败）
  isDestructive?(input): boolean
  validateInput?(input, ctx): Promise<ValidationResult>
  checkPermissions(input, ctx): Promise<PermissionResult>
  maxResultSizeChars: number
}
```

`buildTool(def)` 在安全失败默认值之上合并定义。每个工具必须显式声明更安全的分类。

## 反常规设计

| 模式 | 原因 |
|------|------|
| `isConcurrencySafe: false` 默认 | 安全失败：假设不安全，除非证明安全 |
| `isReadOnly: false` 默认 | 安全失败：假设有写操作，除非证明只读 |
| TeamCreateTool / SendMessageTool 延迟加载 | 避免循环依赖链 |
| REPL 模式下隐藏 Glob/Grep | REPL 已封装它们；暴露两者会造成混淆 |
| 工具 schema 缓存（toolSchemaCache.ts） | Schema 在提示缓存位置 2；任何变更都会使 11,000 Token 失效 |
| 工具定义中的 `strict?: boolean` | 向 API 发送严格 JSON schema 模式以收紧验证 |
