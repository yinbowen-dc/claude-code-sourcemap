# 桥接 / 远程控制基础设施

## 概述

"Bridge"（桥接）是 Claude Code 的远程控制系统：允许外部服务器（Anthropic 云端或 BYOC 环境）
将会话排队并分发给本地的 Claude Code 实例执行。

**相关文件：** `restored-src/src/bridge/`（32 个文件）+ `cli/transports/`（7 个文件）

## 两种实现

### v1：replBridge（基于环境，多工作者）

```
服务器（Environments API）
  ↓  轮询等待任务
replBridge（replBridge.ts）
  ↓  获取容量槽
  ↓  取任务包（workSecret）
  ↓  交换传输层（SSE ↔ WebSocket）
  ↓  运行会话
  ↓  释放容量
```

核心特点：
- **多工作者：** 多个 Claude 实例可以轮询同一队列
- **容量槽：** `capacityWake` 防止超额订阅
- **传输层交换：** 可在保持序列号的情况下将 SSE 迁移到 WebSocket

### v2：remoteBridgeCore（直接 OAuth→JWT）

```
用户 → OAuth 流程 → access_token
  ↓  换取会话 JWT
remoteBridgeCore
  ↓  建立传输层
  ↓  运行会话
  ↓  提前 5 分钟主动刷新 JWT
```

更简单：无轮询、无容量管理、单会话生命周期。

## 核心组件

### FlushGate（刷新门，flushGate.ts）

防止初始历史 POST 期间消息交错。

```typescript
状态机：
  start()      → 'active'：新写入进入队列
  end()        → 返回队列中的待发送项
  deactivate() → 'inactive'：清除标志但不丢弃（传输层交换时使用）
  drop()       → 丢弃队列中的所有项（永久关闭）
```

为什么不用 `Promise.all()`？消息必须严格按 `[历史…, 实时…]` 顺序到达服务器。
`deactivate()` vs `drop()` 的区别：传输层交换时清除标志，但队列消息需要在新传输层上重放。

### capacityWake（容量唤醒，capacityWake.ts）

合并两个中断信号的共享轮询原语：
- **外部信号：** 关闭信号
- **容量控制器：** 容量释放时手动调用 `wake()`

```typescript
// 轮询循环休眠，当容量释放时提前唤醒
await capacityWake.wait(signal)
```

### workSecret（workSecret.ts）

来自 Environments API 的 base64url 编码 JSON：

```typescript
{
  session_ingress_token: string,   // 此会话的认证 Token
  api_base_url: string,            // API 调用目标地址
}
// CCR v2 兼容：将 session_uuid / cse_uuid 规范化为统一格式
```

### trustedDevice（可信设备，trustedDevice.ts）

为桥接会话提供设备级提权 Token：

- 存储在 Keychain 中，90 天滚动有效期
- 作为 `X-Trusted-Device-Token` 请求头发送
- 防止会话重放攻击
- **注册窗口：** 仅在账号创建后 10 分钟内可注册（服务器强制）
- 环境变量覆盖：`CLAUDE_CODE_TRUSTED_DEVICE_TOKEN`（企业级注入）
- GrowthBook 开关：`tengu_sessions_elevated_auth_enforcement`

### jwtUtils（jwtUtils.ts）

```typescript
// 不验证签名地解码（Token 来自可信服务器）
decodeJwtPayload(token: string): JwtPayload

// 主动刷新调度器
createTokenRefreshScheduler():
  - 在过期前 5 分钟调度刷新
  - 每次 schedule() 调用都递增"世代计数器"，使过期的异步刷新失效
  - 每 30 分钟追加重新调度作为安全网
```

### bridgePointer（桥接指针）

崩溃恢复文件：`~/.claude/projects/<dir>/bridge-pointer.json`

- 包含 `sessionId + environmentId`
- 可在程序崩溃后存活
- 4 小时 TTL（基于文件 mtime，非内嵌时间戳）
- Worktree 感知：在错误的 worktree 中 `--continue` 仍能恢复

## 传输层对比

| 传输层 | 方向 | 序列号恢复 | 用于 |
|--------|------|------------|------|
| **SSETransport** | 只读 | `Last-Event-ID` 请求头 | v2 (remoteBridgeCore) |
| **WebSocketTransport** | 双向 | `X-Last-Request-Id` | v1 (replBridge) |
| **HybridTransport** | WS 读 + POST 写 | 通过 WS | v1 (replBridge) |
| **CCRClient** | 仅 HTTP 写 | `event_id` 参数 | v2 (remoteBridgeCore) |

**HybridTransport** 使用 `SerialBatchEventUploader` 处理写入：
- 串行发送保证 `[历史…, 实时…]` 的顺序
- 默认无限重试；replBridge 从外部限制次数

**CCRClient：** 返回类型为 `never`——收到 `409 Conflict` 时无条件抛出异常。

### SerialBatchEventUploader

批量聚合事件，串行发送（同时只有一个在途请求）：

- 默认无限重试（可配置最大次数）
- `replBridge` 限制重试次数以防止僵尸会话
- 打印模式不限制（一次性发送）

## 入站消息处理

### inboundMessages.ts

- 从桥接载荷中提取并规范化用户消息
- 修复 camelCase `mediaType` → `media_type`（来自 iOS 客户端的格式问题）

### inboundAttachments.ts

- 通过 OAuth 将 `file_uuid` 引用解析为本地 `@file` 路径
- **尽力而为：** 失败不会中断消息流
- 文件内容从 Anthropic Files API 获取

## 反常规设计

| 模式 | 原因 |
|------|------|
| 传输层交换时调用 `flushGate.deactivate()`（而非 `drop()`） | 保留队列消息以便在新传输层上重放 |
| 双重 UUID 去重集合（2000 LRU + 无界备用） | 长会话中回显抑制的去重 |
| 每次 `schedule()` 递增世代计数器 | 使过期的异步定时器回调失效 |
| `authRecoveryInFlight` 在第一个 await 前同步声明 | 防止笔记本电脑唤醒时重复 `/bridge` 请求 |
| 归档在 `transport.close()` 之前 | 给上传器提供排空时间（归档是异步的） |
| 桥接指针 TTL 通过文件 mtime 计算 | 无内嵌时间戳；mtime 本身就是 TTL |
| CCRClient 409 → 返回类型 `never` | 类型系统保证：409 之后不会有代码执行 |
