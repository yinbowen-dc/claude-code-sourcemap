# 远程传送、Worktree 与协调器

## Teleport：远程会话编排

### 概述

Teleport 是 Claude Code 的远程执行系统——管理带有 git 上下文、环境配置和文件管理的持久云端会话。

**相关文件：** `utils/teleport.tsx`、`utils/teleport/api.ts`、
`utils/teleport/environments.ts`、`utils/teleport/environmentSelection.ts`、
`utils/teleport/gitBundle.ts`

### 会话架构

```typescript
type SessionResource = {
  type: 'session'
  id: string
  title: string | null
  session_status: 'requires_action' | 'running' | 'idle' | 'archived'
  environment_id: string
  session_context: SessionContext
}

type SessionContext = {
  sources: SessionContextSource[]       // Git 仓库、知识库
  cwd: string
  outcomes: Outcome[] | null            // Git 仓库结果
  custom_system_prompt: string | null
  append_system_prompt: string | null
  model: string | null
  seed_bundle_file_id?: string          // 预置 git 状态
  github_pr?: { owner: string; repo: string; number: number }
  reuse_outcome_branches?: boolean
}
```

### API 函数

```typescript
fetchCodeSessionsFromSessionsAPI()          // 列出远程会话
sendEventToRemoteSession(sessionId, msg)    // 向会话发送消息
updateSessionTitle(sessionId, title)        // 重命名会话
fetchSession(sessionId)                     // 获取单个会话详情
getBranchFromSession(session)               // 从结果中提取 git 分支
```

使用 Anthropic Beta API 请求头：`anthropic-beta: ccr-byoc-2025-07-29`

**重试策略：** 4 次重试，指数退避（2s、4s、8s、16s）。
5xx 时重试，4xx 时立即失败。

### 环境提供商

```typescript
type EnvironmentKind = 'anthropic_cloud' | 'byoc' | 'bridge'
```

- **anthropic_cloud：** Anthropic 管理的云端计算
- **byoc（自带计算）：** 客户管理的计算资源
- **bridge：** 外部桥接连接

系统在没有环境时会**自动创建**默认云端环境。
Bridge 环境默认被过滤掉（作为次级/备用选项）。

## gitBundle：仓库状态传输

### 用途

将本地仓库状态传送到远程会话——把未提交的工作传输到云端。

### 三级降级策略

```
级别 'all'：       git bundle create --all
                   （所有分支、标签、完整历史）
                   ↓ 如果太大
级别 'head'：      git bundle create HEAD
                   （当前分支，完整历史）
                   ↓ 如果还是太大
级别 'squashed'：  git commit-tree（无父提交）
                   （HEAD 树的快照，零历史）
```

### WIP（进行中的工作）捕获

```typescript
// 1. 创建悬空暂存提交（不触碰 refs/stash）
const wipStashSha = await execFile(git, ['stash', 'create'])

// 2. 让其可以被 bundle 引用
await execFile(git, ['update-ref', 'refs/seed/stash', wipStashSha])

// 3. 用 --all bundle（包含 refs/seed/stash）
// 4. 清理：删除 refs/seed/stash（不污染用户的仓库）
```

### 上传

```typescript
uploadFile(bundlePath, '_source_seed.bundle', config)
// 返回：{ fileId, size, scope, hasWip }
// sessionContext.seed_bundle_file_id = fileId
```

通过 `tengu_ccr_bundle_upload` 事件追踪：outcome、size、scope、hasWip、maxBytes。

## Worktree 模式

### 概述

为每个 Agent 任务在 `.claude/worktrees/` 中创建隔离的 git 分支。

**相关文件：** `utils/worktree.ts`、`utils/worktreeModeEnabled.ts`

### 无条件启用

```typescript
// 之前通过 GrowthBook 'tengu_worktree_mode' 控制，但 CACHED_MAY_BE_STALE 模式
// 在缓存填充前的首次启动返回 false，静默丢失 --worktree 标志。
// 现在始终启用。
export function isWorktreeModeEnabled(): boolean { return true }
```

### Slug → 路径 → 分支 映射

```typescript
// "user/feature" → 扁平化 → "user+feature"
worktreeBranchName("user/feature") === "worktree-user+feature"
worktreePathFor(root, "user/feature") === ".claude/worktrees/user+feature"
```

路径验证防止遍历攻击：
- 不允许 `..` 或 `.` 段
- 每段只允许 `[a-zA-Z0-9._-]`
- 总长度最多 64 字符

### 恢复优化

```typescript
// 快速恢复：直接读取 .git 指针文件（避免约 14ms 的 git 子进程）
const existingHead = await readWorktreeHeadSha(worktreePath)
if (existingHead) {
  return { worktreePath, existed: true }  // 跳过获取和初始化
}
```

### 创建后初始化

1. **传播设置：** 复制 `settings.local.json`（可能含有密钥）
2. **配置 git hooks：** `core.hooksPath` 指向主仓库的 hooks
3. **符号链接大型目录：** `settings.worktree.symlinkDirectories`（节省磁盘）
4. **复制 gitignore 文件：** 通过 `.worktreeinclude`（选择性复制）
5. **安装提交归属 hooks：** 如果 `feature('COMMIT_ATTRIBUTION')` 启用

### 大型仓库优化（稀疏检出）

```typescript
const sparsePaths = settings.worktree?.sparsePaths
if (sparsePaths?.length) {
  // 先 --no-checkout，然后：
  git(['sparse-checkout', 'set', '--cone', '--', ...sparsePaths])
  git(['checkout', 'HEAD'])  // 只检出指定路径
}
```

### .worktreeinclude 智能展开

```typescript
// git ls-files --others --ignored --directory
// --directory 折叠完全被 gitignore 的目录：
// 500,000 条目/7 秒 → ~100 条目/100 毫秒

// 智能展开：只展开 .worktreeinclude 模式有明确前缀的目录
const dirsToExpand = collapsedDirs.filter(dir => {
  return patterns.some(p => {
    // 字面量前缀匹配
    if (normalized.startsWith(dir)) return true
    // glob：提取第一个通配符前的字面量前缀
    const globIdx = normalized.search(/[*?[]/)
    if (globIdx > 0) {
      const literalPrefix = normalized.slice(0, globIdx)
      return dir.startsWith(literalPrefix)
    }
    return false
  })
})
```

### 规范根目录 vs 本地根目录

```typescript
// Agent worktree 使用 findCanonicalGitRoot（而非 findGitRoot）
// 即使从会话 worktree 内部生成，Agent worktree 也会落在
// 主仓库的 .claude/worktrees/ 中，而不是
// .claude/worktrees/feature/.claude/worktrees/（不会嵌套）
// 防止嵌套；清理扫描从规范根出发能找到所有 worktree
```

### 基于 Hook 的 VCS 抽象

```typescript
// 优先尝试基于 Hook 的 VCS（允许 Mercurial、Perforce 等）
if (hasWorktreeCreateHook()) {
  const hookResult = await executeWorktreeCreateHook(slug)
  currentWorktreeSession = { ..., hookBased: true }
} else {
  // 回退到 git worktree
  if (!gitRoot) throw new Error('不在 git 仓库中且没有配置 WorktreeCreate hooks')
}
```

### 会话分叉（branch.ts）

通过复制记录来分叉对话：
- 仅复制主对话（排除旁支）
- **保留内容替换元数据**（防止提示缓存预算不匹配）
- 唯一分叉名称生成（Branch → Branch 2 → Branch 3…）

## 协调器模式（Coordinator）

### 概述

协调器 Agent 生成并管理并行工作者，执行复杂的多步骤任务。

**相关文件：** `coordinator/`

### 激活方式

```typescript
isCoordinatorMode(): process.env.CLAUDE_CODE_COORDINATOR_MODE === '1'
// 特性开关：feature('COORDINATOR_MODE')
```

### 协调器系统提示

当 `isCoordinatorMode()` 时注入：

```
角色：跨多个工作者编排软件工程任务。
可用工具：Agent（生成）、SendMessage（继续）、TaskStop（终止）。
工作者可用工具：Bash、Read、Edit、Glob、Grep + MCP 工具。

阶段：
  1. 研究（并行工作者）
  2. 综合（协调器：你阅读发现并理解它们）
  3. 实现（工作者，每次一个文件集）
  4. 验证（工作者）

关键规则：不要写"根据你的发现"——通过包含具体文件路径、行号、确切改动来证明你理解了。
```

### 工作者任务通知

工作者结果以 `<task-notification>` XML 格式出现在用户角色消息中：

```xml
<task-notification>
  <task-id>{agentId}</task-id>
  <status>completed|failed|killed</status>
  <summary>{可读状态摘要}</summary>
  <result>{Agent 的最终文本}</result>
  <usage>
    <total_tokens>N</total_tokens>
    <tool_uses>N</tool_uses>
    <duration_ms>N</duration_ms>
  </usage>
</task-notification>
```

### 继续 vs 重新生成决策矩阵

| 情况 | 使用 |
|------|------|
| 工作者已研究了需要编辑的确切文件 | 继续（SendMessage） |
| 研究范围广，实现范围窄 | 重新生成 |
| 纠正失败 / 扩展最近工作 | 继续 |
| 验证另一个工作者写的代码 | 重新生成（全新视角） |
| 方法完全错误 | 重新生成（干净重来） |
| 完全不相关的任务 | 重新生成 |

### 会话模式匹配

```typescript
// 恢复会话时：如果协调器模式不匹配，翻转环境变量
matchSessionMode(sessionMode: 'coordinator' | 'normal'):
  if (currentIsCoordinator !== sessionIsCoordinator) {
    process.env.CLAUDE_CODE_COORDINATOR_MODE = sessionIsCoordinator ? '1' : undefined
    logEvent('tengu_coordinator_mode_switched', { to: sessionMode })
  }
```

## 反常规设计

| 模式 | 原因 |
|------|------|
| Teleport 自动创建默认云端环境 | 没有环境的用户仍然应该能正常工作 |
| Bridge 环境被默认过滤 | Bridge 是次级/备用，不是首选 |
| gitBundle 三级降级 | 处理任意大小的仓库：完整历史 → 仅 HEAD → 快照 |
| WIP 通过 `stash create`（非 `stash push`）捕获 | 不触碰 `refs/stash`，防止污染用户的暂存列表 |
| Worktree 模式始终启用（曾由 GrowthBook 控制） | 陈旧缓存默认值导致标志被静默丢失——始终启用更好 |
| Agent worktree 使用规范根目录 | 防止嵌套在会话 worktree 内；清理能找到所有目录 |
| gitignored 列表使用 `--directory` 标志 | 500,000 条目/7 秒 → 100 条目/100 毫秒（大型仓库） |
| 智能 glob 展开 | 只展开匹配模式的目录——避免重新遍历大型目录 |
| 协调器的"证明你理解了"规则 | 防止懒惰委托（"根据你的发现……"） |
| 工作者仅工具过滤 | 协调器模式减少工具集，防止工作者再次生成协调器 |
