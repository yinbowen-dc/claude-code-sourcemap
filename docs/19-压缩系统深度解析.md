# 压缩系统深度解析

## 概述

Claude Code 的对话压缩系统是一个多层次的智能系统，包括自动断路器、
基于时间的记忆压缩，以及精心设计的系统提示优化。

**相关文件：** `services/compact/`、`commands/compact/`、`services/api/promptCacheBreakDetection.ts`

---

## 三层压缩架构

```
第 1 层  microcompact（微压缩）
         - 在完整压缩前预处理
         - 移除工具调用的原始输出（保留摘要）
         - 目标：减少传给摘要模型的 token 数量

第 2 层  API microcompact
         - 使用快速模型（Haiku）生成工具调用摘要
         - 比 microcompact 更智能，但更贵
         - 作为 microcompact 的升级替代

第 3 层  full compact（完整压缩）
         - 对整个对话生成结构化摘要
         - 最贵但最准确
         - 结果注入为新的"上下文摘要"消息
```

## autoCompact 断路器

### 设计原因

```typescript
// autoCompact 在上下文即将耗尽时自动触发
// 但如果压缩本身失败（API 超时、模型错误等）
// 会进入循环：触发压缩 → 失败 → 再次触发 → 再次失败

// 解决方案：断路器模式
// 连续失败 3 次后，自动关闭 autoCompact
// 用户需要手动运行 /compact 重置
```

### 实现

```typescript
type AutoCompactState = {
  consecutiveFailures: number
  isCircuitOpen: boolean  // true = 断路，停止自动压缩
  lastFailureTime: number
}

// 每次 autoCompact 失败：
state.consecutiveFailures++
if (state.consecutiveFailures >= 3) {
  state.isCircuitOpen = true
  // 通知用户：自动压缩已暂停
}

// 手动 /compact 成功后：
state.consecutiveFailures = 0
state.isCircuitOpen = false
```

### 📊 实际影响

```
断路器上线前：autoCompact 失败循环每天浪费约 250,000 次 API 调用
断路器上线后：循环立即停止，节省约 250,000 次/天
```

## 基于时间的记忆压缩（Time-Based MC）

### 服务器缓存 TTL 对齐

```typescript
// Anthropic API 的提示缓存（prompt cache）TTL = 1 小时
// 如果会话超过 1 小时，缓存过期，下次请求需要重建

// time-based MC 的触发逻辑：
// 不是"对话太长了就压缩"
// 而是"距上次压缩已经超过 1 小时，即将缓存过期时压缩"

// 这样压缩后建立的新缓存恰好在下一个 1 小时周期开始
// 最大化缓存命中率
const CACHE_TTL_ALIGNMENT = 60 * 60 * 1000  // 1 hour in ms
```

## NO_TOOLS_PREAMBLE：移除工具列表的背后故事

```typescript
// compact 的系统提示中加入 NO_TOOLS_PREAMBLE
// 表示：压缩摘要时不需要工具定义

// 背景：
// 正常对话：工具定义占用大量 token（100-200 个工具 × 每个几百 token）
// 压缩任务：只需要总结对话内容，不需要任何工具

// 实验数据（Haiku 模型）：
// 有工具定义时：2.79% 的摘要请求意外调用了工具
// 移除工具定义后：0.01%
// 工具定义的存在本身会诱导模型去调用它们

// 副作用：节省了每次压缩约 10,000-50,000 token
```

## promptCacheBreakDetection：提示缓存断裂检测

### 50+ 字段 Hash 计算

```typescript
// 系统提示由 50+ 个字段构成
// 任何字段变化都会导致缓存断裂（cache miss）

// 两个独立的 hash：
cacheControlHash   // 哈希"结构性"字段（影响缓存是否命中）
systemHash         // 哈希所有字段（用于变更追踪）

// 为什么分开？
// cacheControlHash 只包含影响缓存的字段
// systemHash 包含所有字段（包括不影响缓存的）
// 这样可以区分：
// - "内容变了但缓存仍然命中"（仅 systemHash 变）
// - "缓存真的断裂了"（cacheControlHash 变）
```

### TTL 过期概率分析

```typescript
// 缓存命中率随时间的概率分布：
// t=0  → 100% 命中（刚建立缓存）
// t=30m → ~80% 命中（30 分钟内通常仍然有效）
// t=55m → ~30% 命中（接近 1 小时，部分服务器已过期）
// t=60m → ~5%  命中（大多数服务器缓存已过期）
// t=65m → ~0%  命中（全部过期）

// 这个分布是因为：
// Anthropic 的服务器不是全部同时过期
// 而是各服务器独立计时（分布式部署）
// 因此缓存命中率是逐渐下降而非断崖式

// 检测逻辑：
// 收到异常低的缓存读取 token 数量时，触发缓存断裂事件
```

## 压缩后操作序列

```typescript
// 压缩完成后的精确操作顺序（顺序很重要）：
// 1. getUserContext.cache.clear?.()
//    清除记忆化的用户上下文缓存
//    原因：压缩后上下文已变，旧缓存失效

// 2. suppressCompactWarning()
//    隐藏"上下文即将耗尽"的警告
//    原因：压缩刚完成，警告不再相关

// 3. notifyCompaction()
//    重置"缓存读取基准"
//    原因：断点缓存检测会比较"压缩前"vs"压缩后"
//          没有这个重置，会误报大量缓存断裂事件

// 4. markPostCompaction()
//    设置标志：下一个请求是压缩后的第一个请求
//    下游系统（如遥测）用此标志过滤异常数据
```

## 反常规设计

| 模式 | 原因 |
|------|------|
| 断路器（3次失败）而非无限重试 | 防止 autoCompact 失败循环浪费 250K API 调用/天 |
| 压缩时机对齐服务器缓存 TTL | 在缓存即将过期时压缩，最大化下一小时的缓存命中率 |
| NO_TOOLS_PREAMBLE | 工具定义存在会诱导 Haiku 调用工具（从 2.79% 降到 0.01%） |
| 先 microcompact 再 full compact | 两步走：减少输入 → 提高摘要质量且更便宜 |
| 分离 cacheControlHash/systemHash | 区分"内容变化"和"缓存断裂"，更精确的分析 |
| 压缩后精确操作顺序 | 每个操作有独立原因；顺序改变会产生误报或资源泄露 |
